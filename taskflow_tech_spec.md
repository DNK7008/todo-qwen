# Техническое задание: Telegram-приложение для управления задачами с фотоотчетами

## 1. Общие сведения
**Наименование проекта:** TaskFlow Telegram App  
**Цель:** Создание веб-приложения в Telegram для постановки, отслеживания задач и передачи фотоотчетов между отделами небольшой компании.  
**Платформа:** Telegram Web App + Bot API  
**Аудитория:** Сотрудники нескольких отделов (до 50 пользователей на старте).

---

## 2. Функциональные требования

### 2.1. Авторизация и регистрация
- **Вход по номеру телефона:**  
  - Автоматическая авторизация через данные Telegram (`user.phone_number` при наличии доступа).  
  - Для пользователей без привязанного номера в Telegram — ручное введение номера с подтверждением через SMS (опционально, требует интеграции с SMS-шлюзом).  
  - Привязка номера к внутреннему ID пользователя в БД.  
- **Роли:**  
  - *Сотрудник:* создание задач, загрузка фото, просмотр своих задач.  
  - *Руководитель отдела:* просмотр всех задач отдела, назначение исполнителей.  
  - *Администратор:* управление отделами, ролями, архивация задач.  

### 2.2. Управление задачами
- **Создание задачи:**  
  - Поля: название, описание, дедлайн, приоритет (низкий/средний/высокий), отдел-исполнитель, ответственный (опционально).  
  - Прикрепление фото (до 5 шт. на задачу) через камеру или галерею устройства.  
- **Статусы задач:**  
  `Новая` → `В работе` → `На проверке` → `Выполнена` / `Отклонена`.  
- **Фильтрация и поиск:**  
  - По отделу, статусу, дате, ответственному.  
  - Теги для категоризации (например, "срочно", "оборудование").  
- **Уведомления:**  
  - О новых задачах, изменении статуса, комментариях (через бота).  

### 2.3. Работа с фото
- Загрузка изображений в задачах (форматы: JPG, PNG, HEIC; макс. размер 10 МБ).  
- Просмотр фото в карточке задачи с возможностью увеличения.  
- Автоматическое сжатие изображений на клиенте для оптимизации трафика.  

### 2.4. Отчетность
- Экспорт задач в CSV/PDF за период (для руководителей).  
- Статистика по отделам: количество выполненных задач, среднее время выполнения.  

---

## 3. Нефункциональные требования
- **Безопасность:**  
  - Шифрование данных в транзите (HTTPS).  
  - Валидация всех входящих данных от клиента.  
  - Разграничение доступа на уровне БД и API.  
- **Производительность:**  
  - Время отклика API < 500 мс при нагрузке до 100 одновременных пользователей.  
  - Поддержка до 1000 задач в активной базе.  
- **Масштабируемость:**  
  - Горизонтальное масштабирование бэкенда через Docker-контейнеры.  
  - Хранение фото в объектном хранилище (S3-совместимом) для легкого расширения.  
- **Совместимость:**  
  - Корректная работа в Telegram Web App на iOS/Android/Desktop.  

---

## 4. Технические требования
### 4.1. Стек технологий
| Компонент       | Технология                          | Обоснование                     |
|-----------------|-------------------------------------|---------------------------------|
| Фронтенд        | React + TypeScript + Telegram SDK   | Быстрая разработка, типизация   |
| Бэкенд          | Python + FastAPI                    | Асинхронность, простота интеграции с TG |
| База данных     | PostgreSQL                          | Надежность, поддержка JSON-полей |
| Хранение фото   | MinIO (S3-совместимое)              | Локальное развертывание, дешевизна |
| Контейнеризация | Docker + Docker Compose             | Упрощение деплоя                |
| Деплой          | VPS (Ubuntu 22.04)                  | Контроль над инфраструктурой    |

### 4.2. Архитектура
```
Telegram Client → [Telegram Bot] → [FastAPI Backend]  
                      ↓                   ↓  
                [PostgreSQL]       [MinIO Storage]  
```
- **Бот:** Обрабатывает команды `/start`, `/tasks`, пересылает данные в Web App.  
- **Web App:** Интерфейс для работы с задачами (открывается через кнопку в боте).  
- **Backend:** REST API для фронтенда, обработка бизнес-логики.  

### 4.3. Интеграции
- **Telegram Bot API:** Для уведомлений и глубоких ссылок на Web App.  
- **SMS-шлюз (опционально):** Twilio/Infobip для подтверждения номера телефона.  

---

## 5. Этапы разработки
1. **Прототип (2 недели):**  
   - Базовая авторизация через Telegram.  
   - Создание/просмотр задач без фото.  
2. **MVP (4 недели):**  
   - Загрузка фото, статусы задач, уведомления.  
   - Ролевая модель (сотрудник/руководитель).  
3. **Расширение (2 недели):**  
   - Отчетность, экспорт данных.  
   - Оптимизация производительности.  
4. **Тестирование и деплой (1 неделя):**  
   - Нагрузочное тестирование.  
   - Развертывание на сервере заказчика.  

---

## 6. Требования к серверу
- **Минимальная конфигурация:**  
  - 2 vCPU, 4 ГБ ОЗУ, 50 ГБ SSD.  
  - ОС: Ubuntu 22.04 LTS.  
- **Домен:** Наличие собственного домена для HTTPS (Let's Encrypt).  
- **Резервное копирование:** Ежедневные бэкапы БД и хранилища фото.  

---

## 7. Дополнительные возможности (будущие версии)
- Чат в рамках задачи для обсуждения деталей.  
- Голосовые заметки как альтернатива тексту.  
- Интеграция с календарями (Google Calendar, Outlook).  
- Мобильное приложение вне Telegram.  

---

## 8. Критерии приемки
- Все функции из раздела 2 реализованы и протестированы.  
- Нагрузка в 50 пользователей не вызывает сбоев.  
- Документация для администраторов и пользователей предоставлена.  
- Исходный код передан заказчику с инструкцией по деплою.  

---
**Примечание:**  
- Для входа по номеру телефона в 95% случаев достаточно данных из Telegram (если пользователь разрешил доступ к номеру в настройках приватности). Ручной ввод с SMS требуется редко.  
- Рекомендуется начать с авторизации через Telegram ID, добавив номер телефона как дополнительный атрибут профиля.
